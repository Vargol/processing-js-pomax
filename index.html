Welcome to a Processing.js fork.

This fork sacrifices processing compatabilty at the alter of faster drawing.

The change makes color variables opaque objects instead of an integer allowing the string conversion needed for using the colour on a html5 canvas to occur on color creation rather than color use. If you then reuse color variables rather than generate them on the fly you can get increases in the speed of sketch by over 50% when the sketch spends most of its time drawing strokes, fills and points.

What you lose is the ability to alter a color value using maths, if you need to do that store the number version.

To prove such outrageous claims in improvement here's an example. Two sketches using the same pde file, one runinng processing.js 1.4.1 and one one runnig my fork.

The processing code is set to stop drawing at a fixed point and show how much work its done and how long it took, I get the follow results.

processing.js 4087: 67328
my fork       4087: 32398

My maths works that out as running in 48.12% of the time the original processing.js version ran it, or 51.88% faster or, in english, over twice as fast.
